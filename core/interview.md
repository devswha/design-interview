# 디자인 인터뷰 프레임워크

deep-interview의 약점-차원 타겟팅을 디자인 도메인에 적용한 6차원 명료도 모델.

## 차원과 가중치

| 차원 | 가중치 | 명료 기준 (1.0) | 모호 신호 (0.0) |
|---|---|---|---|
| audience | 0.20 | 도착 경로 + 도착 시점의 인지 상태가 한 문장으로 기술됨 | "일반 사용자요" |
| mood | 0.20 | 형용사 2–3개 + 대비 진술 ("○○하지만 ○○하지 않은") | "깔끔하게요" |
| brand | 0.10 | 기존 자산 유무 확정, 있으면 색/서체 확보 | 자산 존재 여부 미확인 |
| structure | 0.20 | 첫 화면 필수 요소가 우선순위로 나열됨 | "알아서 배치해주세요" |
| conversion | 0.20 | 단 하나의 전환 행동 + CTA 문구 방향 확정 | 전환 목표 복수 또는 부재 |
| reference | 0.10 | 실제 URL/제품 1개 이상 + 좋았던 이유 | 레퍼런스 없음 (greenfield면 0.5에서 시작) |

전체 점수 = Σ(차원 점수 × 가중치). **명료도 임계값: 0.80** (`--quick`은 audience/mood/conversion 3차원 평균 0.75).

## 라운드 운영

1. 라운드 시작 시 표 형태로 현재 점수를 보여준다.
2. 가장 약한 차원을 지명하고, 왜 그 차원을 묻는지 한 줄로 말한다.
3. 질문은 하나. **기본은 객관식** — AskUserQuestion으로 3–5개 선택지 + 항상 "직접 입력 / 없음·알아서" 폴백을 준다. 선택지는 *디자이너의 제안*이다: 안전한 디폴트로 수렴시키지 말고 **서로 다른 방향**을 제시해, 하나를 고르면 관점이 잠기게 한다(같은 메뉴를 매번 같은 순서로 주지 않는다 — 수렴이 곧 AI 티다). 사용자의 자유 입력이 언제나 선택지를 이긴다.
4. 답변 반영 후 점수 갱신. 한 답변이 여러 차원을 올릴 수 있다.

## 차원별 질문 설계 요령

- **audience**: 인구통계가 아니라 *도착 맥락*을 묻는다. "이 페이지 링크를 어디서 누르고 들어오나요?"
- **mood**: 형용사를 직접 묻지 말고 대비를 강제한다. "고급스러움과 친근함 중 하나를 포기해야 한다면?"
- **brand**: 존재 확인 → 자산 수집 순서. 자산이 있으면 그 이상 묻지 않는다.
- **structure**: "스크롤 없이 보이는 화면에 단 세 가지만 남긴다면?"
- **conversion**: 전환이 복수로 나오면 즉시 단수화를 요구한다. "딱 하나만 측정한다면 어떤 클릭인가요?"
- **reference**: `--deep`에서만 좋은 예/싫은 예 양쪽을 수집. 싫은 예가 design-tells 우선순위를 정해준다.

## 차원별 선택지 예시 (AskUserQuestion)

선택지는 메뉴가 아니라 *제안*이다 — 소스와 앞선 답에 맞춰 매번 새로 짜고, 항상 "직접 입력"을 포함한다. 무드/레퍼런스가 'AI 디폴트(따뜻한 크림+어스톤+시스템 산스)' 탈출의 본진이므로 가장 공들여 다양하게 준다.

- **mood** (최우선): 추상 형용사 말고 서로 다른 *가게/장소 아키타입*으로 — 예: `잘 정돈된 일본 문구점(절제·격자)` / `손때 묻은 동네 책방(따뜻·세리프)` / `개발자 터미널(다크·모노·기능)` / `미니멀 갤러리(여백·대형 타입)` / `90s 인쇄 잡지(그리드 깨기·강대비)`. 별도 질문으로 **"이렇게는 절대 안 됨" 하나**도 받는다(싫은 게 방향을 더 빨리 잡는다).
- **reference**: ① `참고 URL 있음 → 주소 입력`(주면 저장·분석 최우선) ② `무드 아키타입에서 고름`(위) ③ `없음, 알아서`. URL은 SSRF 가드 fetch + desktop/mobile 스크린샷으로 보존하고 "빌릴 것 / 버릴 것 / 복제 금지" 브리프에 적는다.
- **audience**: 도착 맥락 — 예: `광고·SNS 보고 처음` / `지인 추천·DM` / `검색하다 발견` / `브랜드 알고 재방문`.
- **structure**: 첫 화면에 남길 요소 3개를 고르게 — 예: 헤드라인 · 핵심 숫자 · 데모/스크린샷 · 신뢰 증거 · CTA · 한 줄 설명 중 셋.
- **conversion**: 단일 전환 선택. 복수가 나오면 즉시 `딱 하나만 측정한다면?`으로 단수화.
- **brand**: `색/서체/로고 있음 → 자산 첨부` / `로고만 있음` / `전혀 없음(자유)`.

## 소스 선반영 규칙

slop 소스에서 이미 추론 가능한 차원은 질문 대신 **확인**으로 처리한다:

> 소스에 "30개 템플릿, 팀 워크플로우" 클레임이 있어 audience를 '협업 툴 도입을 검토 중인 팀 리드'로 추정합니다. 맞나요?

확인은 1라운드에 여러 개 묶어도 된다 (질문이 아니라 검증이므로).

## 제안서 장르 보정 (`--page proposal`)

제안서는 conversion이 사전 해결된 장르(승인 또는 다음 미팅)라 차원의 *해석*이 바뀐다.
정본은 `design-principles.md`의 "제안서 인터뷰 보정" — 요지:

| 차원 | 제안서에서 묻는 것 |
|---|---|
| audience | 누가 어떤 루브릭으로 채점하는가 — "평가기준표/배점표가 있나요?" RFP가 있으면 배점은 소스 선반영 확인으로 |
| mood | 권위↔친근 트레이드오프를 문서 질감으로 매핑 |
| brand | 클라이언트의 존재감이 벤더보다 우선 — 수신 표지 헤더의 클라이언트명 크기까지 |
| structure | 첫 화면이 아니라 평가자가 가장 박하게 채점하는 섹션. 섹션 분량 예산은 배점을 따른다 (10점당 ~1.5–2쪽) |
| conversion | 사전 해결 — 유일 CTA(승인/미팅/서명)의 형태와 실명 담당자만 확정 |
| reference | 이 관계에서 과거에 이기고 진 제안서 |

가중치는 동일하게 유지하되, conversion은 보통 첫 라운드에 1.0으로 잠긴다.
